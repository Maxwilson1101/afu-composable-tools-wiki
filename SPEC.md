# Problem Trace System — SPEC v1.0

A **problem trace system** is a structured, agent-operable collection of Markdown files that records how large problems are decomposed, solved, critiqued, and transferred across domains.

---

## 1. Overview

### Definitions

| Term | Replaces | Cargo Analogue | Definition |
|---|---|---|---|
| **Trace** | Project | workspace / package | A full problem-solving record for one large, meaningful problem |
| **Module** | Crate | crate | A major logical phase of a Trace; owns a set of Units |
| **Unit** | Problem Unit | module (source file) | The smallest reusable problem-solving component |

### Design Rationale

The three terms form a consistent family — each names a structural level of problem-solving knowledge, not a software artifact. A **Trace** records *how a problem was solved*. A **Module** records *one major phase of that solution*. A **Unit** records *one atomic subproblem*.

### Directory Structure

```text
<trace-name>/
│
├── .log/
│   └── agent.ndjson              # Machine-readable operation log (NON-MARKDOWN)
│
├── trace.md                      # Trace index — required
│
├── <module-name>/                # One folder per Module
│   ├── module.md                 # Module index — required per module
│   └── <unit-name>.md            # Unit files
│
├── assets/                       # Images, SVGs, figures (flat, no subfolders)
│   └── *.svg / *.png
│
└── prompts/                      # Temporary agent prompts, outputs
│   ├── <prompt-name>.md          # Agent input prompts
│   └── <output-name>.md          # Agent-generated outputs
│
├── CRITIQUE.md                   # Trace-wide critique generated by Agent
└── DERIVED.md                    # Derived problems generated by Agent
```

### Directory Rules

1. The root file is named `trace.md`.
2. Each module folder must contain a `module.md` index file.
3. `assets/` is flat — no subfolders.
4. `AI/` contains only temporary prompts and outputs for future Skill Extraction.
5. `CRITIQUE.md` contains trace-wide critiques
6. `DERIVED.md` holds for agent generated problems
7. `.log/` is reserved for the log system — no Markdown files allowed inside it.

## 2. Markdown Schema

### 2.1 Trace Index

#### Schema

```markdown
---
problem: <string>
created: <YYYY-MM-DD>
---

## Modules

- [[module-name]] @TODO
- ...

## Critique

[[CRITIQUE]]

## Derived Problems

[[DERIVED]]
```

#### Overview

Each problem record contains:

- Core metadata describing the problem
- A list of associated modules in order
- A link to critique records file
- A link to derived problems file

#### Metadata

##### TODO Marker

The `@TODO` marker indicates that:

- The referenced module **exists but is not yet completed**, or
- The module **requires further processing**, refinement, or validation.

Modules without the `@TODO` marker are considered:

- Completed
- Reviewed
- Ready for reuse

This mechanism enables **incremental workflow management** and supports tracking unfinished work.

#### Example

```markdown
---
problem: How to extract eeg features?
created: 2026-04-15
---

## Modules

- [[build-dataset-pytorch]]
- [[build-dataset-grain]]
- [[visualize-matplotlib]]
- [[visualize-strategy]]
- [[visualize-plotly]]
- [[visualize-evaluation]] @TODO
- [[research-eeg-feature-engineering]] @TODO

## Critique

[[CRITIQUE]]

## Derived Problems

[[DERIVED]]
```

### 2.2 Module Index

#### Schema

```markdown
---
problem: <string>
created: <YYYY-MM-DD>
---

## Module Deps

- [[model-name]]
- ...

## Units

- [[unit-name]] @TODO
- ...

## Critique

module-wide critique generated by Agent

```

#### Overview

Each module record contains:

- Core metadata describing the module-level problem context
- A list of dependent modules required for this module
- A list of associated problem units in execution order
- A module-wide critique section generated by the agent

The **Module Deps** section defines the dependency relationships between modules.  
These dependencies specify which modules must be available or completed before the current module can be executed.

The **Units** section represents the ordered sequence of problem units that compose the internal workflow of the module.

The **Critique** section records **module-wide observations and issues** generated during review. Unlike the trace-wide critique log, which captures system-level feedback, this section focuses on **local structural or logical issues within a single module**.

#### Metadata

##### TODO Marker

The same as TODO Marker in Trace index

#### Example

```markdown
---
problem: How to visualize an EEG dataset across multiple dimensions?
created: 2026-04-15
---

## Module Deps

- [[build-eeg-dataset]]

## Units

- [[...]]

## Critique

...
```

### 2.3 Unit File

#### Schema

```markdown
---
problem: <string>
created: <YYYY-MM-DD>
tags:
- tag 1
- tag 2
---

## Deps

- [[unit-path]]
  reason

## Attempt [Brief Method Name]

### Problem [Concise Name] @field

Overview:

problem overview including relevant code and execution traces

Fix: @AI @Verified

detailed records describing the steps taken to resolve the problem

### Question [Concise Name] @field

Description:

detailed description of the specific question being addressed

Answer: @AI @Verified

final answer or resolution to the question

## Decision (Optional)

final decision made by user

## Critique (Optional)

unit-wide critique
```

#### Overview

This schema defines the structure of a **problem unit record**, which captures dependency context, solution attempts, problem–question pairs, and unit-level evaluation.

Each unit record is designed to support **traceable reasoning**, **iterative problem-solving**, and **structured refinement** of solutions.

A unit consists of:

- Core metadata describing the problem context
- Dependency references to prerequisite units
- One or more solution attempts
- A final decision when multiple attempts exist
- A unit-wide critique section for evaluation and refinement

This structure enables:

- Explicit dependency tracking
- Iterative method exploration
- Verified solution recording
- Final strategy selection

#### Metadata

##### TODO Marker

TODO Marker is also available in units.

##### Bracked Concise name

Each **Problem** and **Question** section includes a bracketed concise name:

The **[Concise Name]** represents:

- A short, descriptive summary of the problem or question
- A human-readable identifier for quick recognition
- A label that captures the essential nature of the issue

The concise name should:

- Be brief and specific
- Avoid full sentences
- Clearly reflect the core challenge

Example:

```
Problem [Null Pointer Crash] @debugging
Problem [Binary Search Boundary Error] @algorithms

Question [Why does recursion overflow?] @programming-languages
Question [How to detect cyclic dependencies?] @graph-theory
```

##### Domain Annotation

The `@field` annotation specifies the **domain or technical area** associated with the problem or question. (Multi fields are supported!)

It provides:

- Domain classification
- Search and filtering capability
- Context awareness during skill extraction

Examples:

- `@algorithms`
- `@systems`
- `@machine-learning`
- `@debugging`
- `@databases`
- `@linear-algebra`

#### Example

```markdown
---
problem: How to partition a labeled array dataset by class?
created: 2026-04-15
tags: 
- python-numpy
---

## Deps

- [[load-npz-file]]
  requires loaded NPZ object as input
- [[extract-npz-data]]
  requires extracted train_data and train_label

```

### 2.5 Trace-wide Critique (CRITIQUE.md)

#### Schema

```markdown
# Trace Critique

## 1. Structural Critique

Focus: Evaluate internal module quality and structural soundness, independent of execution order.

### Checklists

- [ ] Module Reusability — Is each module designed to be reusable across different tasks or contexts?
- [ ] Unit Conflict Detection — Do units inside the module overlap, duplicate responsibility, or conflict?
- [ ] Unit Generalization — Can similar units be abstracted into a shared or parameterized form?
- [ ] Module Naming Clarity — Does the module name clearly reflect its responsibility and boundaries?
- [ ] Module Responsibility Scope — Does the module avoid mixing unrelated concerns?

### Issues

#### Issue <N> — Module Reusability Limitation

**Problem:**

A module is tightly coupled to a specific task, dataset, or context, limiting reuse.

**Why it matters:**

Low reusability increases duplication, maintenance cost, and inconsistency across workflows.

**Suggested improvement:**

- Introduce parameterization.
- Extract context-specific logic into configurable inputs.
- Refactor shared logic into reusable sub-units.

#### Issue <N> — Unit Conflict or Overlap

**Problem:**

Multiple units perform overlapping responsibilities or produce conflicting outputs.

**Why it matters:**

Conflicts increase ambiguity, reduce maintainability, and complicate downstream composition.

**Suggested improvement:**

- Merge redundant units.
- Introduce a shared abstraction layer.
- Separate responsibilities more cleanly.

#### Issue <N> — Unit Generalization Opportunity

**Problem:**

Similar units exist with minor variations but are not abstracted.

**Why it matters:**

Missed generalization opportunities increase system complexity and redundancy.

**Suggested improvement:**

- Replace repeated logic with a generalized unit.
- Introduce parameterized patterns.
- Define reusable templates.

#### Issue <N> — Module Naming Ambiguity

**Problem:**

Module names are vague, overly generic, or misleading.

**Why it matters:**

Poor naming increases cognitive load and reduces system readability.

**Suggested improvement:**

- Rename using verb-object or function-intent format.
- Ensure names reflect outputs and responsibility boundaries.
- Avoid overloaded terminology.

#### Issue <N> — Mixed Responsibility Module

**Problem:**

A module handles multiple unrelated concerns.

**Why it matters:**

Mixed responsibilities reduce modularity and hinder reuse.

**Suggested improvement:**

- Split module into smaller focused modules.
- Align each module to a single responsibility.
- Define clear input/output boundaries.

---

## 2. Workflow Critique

Focus: Evaluate how modules are combined to accomplish a task.

### Checklists

- [ ] Alternative Composition — Could the same goal be achieved with fewer modules?
- [ ] Composition Efficiency — Are redundant steps present in the workflow?
- [ ] Module Reordering — Would reordering modules simplify the workflow?
- [ ] Missing Composition — Are useful modules absent that could simplify logic?
- [ ] Over-Specialization — Are highly specific modules used where generic ones would suffice?
- [ ] Composability — Do modules connect cleanly via compatible inputs/outputs?

### Issues

#### Issue <N> — Suboptimal Module Composition

**Problem:**

The workflow uses a sequence of modules that is longer or more complex than necessary.

**Why it matters:**

Inefficient composition increases runtime, maintenance complexity, and cognitive load.

**Suggested improvement:**

- Replace chains with higher-level modules.
- Merge compatible modules.
- Reduce unnecessary intermediate outputs.

#### Issue <N> — Redundant Workflow Steps

**Problem:**

The workflow repeats equivalent logic or unnecessary transformations.

**Why it matters:**

Redundancy increases fragility and execution overhead.

**Suggested improvement:**

- Remove duplicate steps.
- Cache reusable results.
- Consolidate repeated operations.

#### Issue <N> — Missing Higher-Level Module

**Problem:**

Multiple low-level modules are used where a higher-level abstraction could exist.

**Why it matters:**

Missing abstraction reduces clarity and increases maintenance cost.

**Suggested improvement:**

- Introduce composite modules.
- Encapsulate repeated module patterns.
- Define reusable workflow templates.

#### Issue <N> — Poor Module Ordering

**Problem:**

Modules are executed in an order that creates unnecessary dependencies or transformations.

**Why it matters:**

Ordering inefficiencies can propagate unnecessary complexity.

**Suggested improvement:**

- Reorder modules to reduce transformations.
- Move normalization or validation earlier.
- Delay expensive operations until necessary.
```

#### Overview

This schema defines a **two-layer critique framework** for evaluating modular systems or execution traces.

This schema follows three core principles:

1. **Local-first correctness**  
    Modules must be structurally sound before workflow optimization.
2. **Composability-first design**  
    Systems should be optimized around reusable building blocks.
3. **Abstraction-driven improvement**  
    Repeated patterns should become reusable modules.

##### Structural Critique — Local Quality

Evaluates **individual modules** in isolation.

Primary goals:

1. Improve **module reusability**
2. Eliminate **unit conflicts**
3. Encourage **unit generalization**
4. Improve **module naming clarity**
5. Enforce **single-responsibility design**

This layer answers:

> "Is each module internally well-designed?"

##### Workflow Critique — Global Composition

Evaluates **how modules are combined** to perform a task.

Primary goals:

1. Reduce workflow complexity
2. Improve composition efficiency
3. Introduce higher-level abstractions
4. Optimize execution ordering
5. Improve composability between modules

This layer answers:

> "Is this the best way to combine these modules?"

#### Example

```markdown
# Trace Critique

## 1. Structural Critique

### Checklists

- [x] Module Reusability — Some modules are reusable, others are task-specific.
- [x] Unit Conflict Detection — Overlapping units detected.
- [x] Unit Generalization — Multiple similar units identified.
- [x] Module Naming Clarity — Naming inconsistencies found.
- [ ] Module Responsibility Scope — Mostly single-purpose modules.

### Issues

#### Issue 1 — Unit Conflict or Overlap

Problem:

The module `TextProcessing` contains both `TokenCleaner` and `TokenNormalizer`, which partially duplicate normalization logic.

Why it matters:

Overlapping logic increases maintenance burden and risks inconsistent outputs.

Suggested improvement:

Merge normalization logic into a shared `TokenNormalizationUnit` and reuse across modules.

---

#### Issue 2 — Module Naming Ambiguity

Problem:

The module name `Handler` does not specify its responsibility.

Why it matters:

Ambiguous names increase cognitive load and reduce trace readability.

Suggested improvement:

Rename `Handler` to `UserInputValidationHandler`.

---

#### Issue 3 — Unit Generalization Opportunity

Problem:

Two units `CSVParser` and `TSVParser` implement nearly identical parsing logic.

Why it matters:

Duplicated logic increases system size and maintenance effort.

Suggested improvement:

Create a generalized `DelimitedFileParser` with configurable delimiter parameter.

---

## 2. Workflow Critique

### Checklists

- [x] Alternative Composition — Fewer modules possible.
- [x] Composition Efficiency — Redundant normalization detected.
- [ ] Module Reordering — Current order mostly valid.
- [x] Missing Composition — Higher-level parser module absent.
- [x] Over-Specialization — Some modules overly specific.
- [x] Composability — Minor interface mismatches exist.

### Issues

#### Issue 1 — Redundant Workflow Steps

Problem:

The workflow applies `TextNormalization` twice before and after parsing.

Why it matters:

Redundant steps increase execution time and risk inconsistent normalization.

Suggested improvement:

Remove the second normalization step and ensure normalization occurs once at input boundary.

---

#### Issue 2 — Missing Higher-Level Module

Problem:

Multiple parsing-related modules are chained together.

Why it matters:

This pattern repeats across workflows.

Suggested improvement:

Introduce a `StructuredTextParsingPipeline` module that encapsulates parsing, validation, and normalization.
```

### 2.6 Trace-wide dervied problems (DERIVED.md)

Valid only inside `ai-output` files with `operation: generate-problems`.

| Keyword | Level | Syntax | Meaning |
|---|---|---|---|
| `Core Patterns` | H2 | `## Core Patterns` | Detected domain-agnostic reasoning patterns |
| `Transfer` | H2 | `## Transfer — <domain>` | Opens a cross-domain problem block |
| `Easy` | H3 | `### Easy` | Concept transfer problems |
| `Medium` | H3 | `### Medium` | Applied reasoning problems |
| `Hard` | H3 | `### Hard` | Cross-domain system design problems |
