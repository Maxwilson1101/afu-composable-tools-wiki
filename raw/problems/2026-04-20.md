# Data Loading with Validation Pipeline

## Problem 1 — debugging

**Scenario.** A colleague on your team ships a small utility that loads a JSON config file for a training run, validates it, and returns a parsed config object. You notice that when the config is malformed, users keep filing tickets saying "it crashed somewhere, I don't know why" — the errors they get are things like `KeyError: 'optimizer'` deep inside the training loop, not at config-load time. You pull up the loader:

```python
def load_config(path):
    cfg = json.load(open(path))
    if not isinstance(cfg["learning_rate"], float):
        raise ValueError("bad config")
    if cfg["optimizer"] not in ["adam", "sgd", "rmsprop"]:
        raise ValueError("bad config")
    if cfg["batch_size"] < 1:
        raise ValueError("bad config")
    return cfg
```

**Task.** Rewrite this loader so the failure mode users are complaining about stops happening, and so that when validation does fail, the ticket tells them exactly what's wrong.

**Constraints.**

- The required keys are `learning_rate`, `optimizer`, `batch_size`, `model_name`.
- `optimizer` must be one of `adam`, `sgd`, `rmsprop`.
- The fix should still be a single function callable as `load_config(path)`.

## Problem 2 — implementation

**Scenario.** You're building a data-loading layer for an image classification project. Images come from three sources — a `train/` split, a `val/` split, and a `test/` split — each stored as a single `.npz` file containing one array of images (shape `(N, H, W, 3)`) and one array of integer labels (0–9). Your teammate has asked for a single function they can call with the split name and get back an iterable of `(image, label)` pairs, validated.

**Task.** Write the loader function.

**Constraints.**

- Callers pass a split name; invalid split names should not reach the file system.
- The loader should refuse to return data if the images array and labels array disagree on `N`, or if any label is outside 0–9, or if the image shape's trailing dimensions aren't `(H, W, 3)` where H and W are whatever the file provides — only the channel count is fixed.
- Caller should get a clear, specific error for each failure mode, not a generic "validation failed."

## Problem 3 — design

**Scenario.** Your team maintains a library that ingests financial tick data from four different exchanges. Each exchange delivers data in a slightly different `.parquet` schema, but conceptually they all produce the same thing: a stream of `(timestamp, symbol, price, volume)` records. A new requirement has come in: downstream consumers need to be able to ask for any exchange's data by name and get back an iterable of validated records, with the validation rules being mostly shared across exchanges but with per-exchange quirks (e.g., exchange A allows negative volume for short trades; exchange B does not).

**Task.** Sketch the structure of the ingestion module — what the public surface looks like, how validation is organized internally, and how you'd add a fifth exchange six months from now without touching existing code.

**Constraints.**

- A new exchange should be addable by writing one new file or one new block, not by editing four existing ones.
- Validation that's shared across exchanges must not be duplicated per exchange.
- The public entry point is a single function that takes an exchange name and returns the validated stream.
